# Chapter 1 - Introduction to Apache OFBiz

---

# What is Apache OFBiz?

Apache OFBiz (Open For Business) is an **Open Source Enterprise Application Framework**.

It is mainly used for building **large business applications**.

Examples:

- Order Management System (OMS)
- Warehouse Management System (WMS)
- Inventory Management
- Product Management
- Manufacturing
- Accounting
- CRM (Customer Relationship Management)

Instead of creating everything from scratch, OFBiz already provides most of the common business modules.

---

# Why was OFBiz created?

Suppose Company A wants to build an Order Management System.

They need:

- Products
- Orders
- Customers
- Inventory
- Payments
- Shipments
- Users
- Security
- Reports

Now imagine Company B also needs the same things.

Again Company C.

Again Company D.

Almost every business software requires these modules.

So instead of writing everything again and again, Apache created OFBiz.

Now developers only need to write the custom business logic.

---

# What does OFBiz provide?

OFBiz already provides many frameworks.

Some important ones are:

- Entity Engine
- Service Engine
- Screen Widget
- Form Widget
- Menu Widget
- Security Framework
- Transaction Management
- Scheduler
- Workflow
- Internationalization (i18n)
- Localization (l10n)

These are already built inside the framework.

---

# What is an Enterprise Framework?

Enterprise means software used by companies to run their business.

Examples:

- Amazon OMS
- Walmart Inventory System
- ERP Software
- Warehouse Software
- Manufacturing Software

OFBiz is specially designed for these kinds of enterprise applications.

# Business Logic

Business Logic means:

"The rules of the business."

Example:

Customer places an order.

Business Logic decides:

- Can the order be approved?
- Is inventory available?
- Which warehouse should fulfill it?
- Has payment been completed?
- Can shipment be created?

These rules are called Business Logic.

---

# Ways to write Business Logic in OFBiz

Business Logic can be written in multiple ways.

## 1. Entity Auto Service

Used for simple CRUD operations.

Examples:

- Create
- Update
- Delete

No Java code required.

---

## 2. Simple Service (XML)

Used for simple workflows using XML actions.

Suitable when the logic is straightforward.

---

## 3. Groovy Service

Most commonly used.

Used when the business logic becomes complex.

Examples:

- Loops
- Conditions
- Entity Queries
- Calling Other Services
- Data Transformations

---

## 4. Java Service

Used for very complex business logic.

Examples:

- Heavy Processing
- External Integrations
- Performance Critical Logic
- Large Business Rules

---

# Chapter 2 - OFBiz Folder Structure & Component Architecture

---

# Why do we need to understand the Folder Structure?

Whenever we download OFBiz, thousands of files are present.

If we don't know which folder is responsible for what, then debugging and development become very difficult.

Understanding the folder structure is like understanding the map of a city.

If you know the map, then you know exactly where to go.

---

# Root Directory Structure

After downloading OFBiz, you'll see something like this:

```
ofbiz/

├── applications/

├── framework/

├── plugins/

├── runtime/

├── gradle/

├── themes/

├── lib/

├── build.gradle

├── settings.gradle

└── gradlew
```

Each folder has its own responsibility.

---

# 1. framework/

The **framework** folder contains the core OFBiz engine.

Everything related to the framework is present here.

Examples:

- Entity Engine
- Service Engine
- Widget Engine
- Security
- Transactions
- Base Utilities
- Common Components

Think of it as the **heart of OFBiz**.

Without the framework, OFBiz cannot run.

---

Inside framework, you'll find folders like:

```
framework/

├── base/

├── entity/

├── service/

├── webapp/

├── widget/

├── security/

├── common/

└── ...
```

---

## framework/entity

Responsible for:

- Entity Engine

- Database Communication
- Entity Definitions
- Delegator
- GenericValue

Important files to study later:

```

Delegator.java

entityengine.xml
```

---

## framework/service

Responsible for:

- Service Engine

Important classes:

```

```

---

## framework/widget

## framework/base

Contains common utilities.

Examples:

```

```

These helper classes are used throughout OFBiz.

---

# 2. applications/

This folder contains the default business applications provided by OFBiz.

- Orders
- Order Services
- Order Screens
- Order Events
- Order Entities

---

# 3. plugins/

This is the most important folder for developers.

Plugins are used for **custom development**.

Whenever a client asks for a custom feature,

we usually create our own component inside plugins.

Example:

```
plugins/

├── example/

├── mycompany/

├── training/

└── customComponent/
```

Instead of modifying OFBiz source code,

we write our custom code here.

This makes upgrades much easier.

---

# Why should we avoid modifying framework?

Suppose you modify

```
framework/entity
```

and later OFBiz releases a new version.

Now your changes may get overwritten.

Instead,

write your custom code inside

```
plugins/
```

This is considered a best practice.

---

# 4. runtime/

This folder is created while OFBiz runs.

It stores runtime-generated files.

Examples:

```
runtime/

├── logs/

├── derby/

├── temp/

├── txlog/

├── upload/

└── cache/
```

Contains log files.

Examples:


Whenever an exception occurs,

the first place to check is:

```
runtime/logs/ofbiz.log
```

---

## runtime/derby

If you're using Derby,

its database files are stored here.

If using MySQL or PostgreSQL,

this folder becomes less important.

---

# 5. themes/

Contains themes used for UI rendering.

Examples:

```

...
```

Themes define:

- Header
- Footer

- Navigation

# 6. gradle/

Responsible for:

- Building Project
- Downloading Dependencies

# 7. build.gradle

This is the Gradle build file.

Responsible for:

- Dependencies
- Plugins

- Packaging

---

# 8. settings.gradle

Contains Gradle project settings.

---

# 9. gradlew

This is the Gradle Wrapper.

Instead of installing Gradle manually,

we simply run:

```bash

```

# Chapter 3 - OFBiz Component Architecture

---

# What is a Component?

A Component is a **self-contained module** inside OFBiz.

You can think of a component as a small application that contains everything related to one business module.

Examples:

- Order
- Product
- Party
- Accounting
- Manufacturing
- Example

Every component has its own:

- Entities
- Services
- Screens
- Forms
- Events
- Java Files
- Groovy Files
- Data
- Configuration

Instead of keeping everything in one place, OFBiz organizes code into Components.

---

# Why Components?

Suppose OFBiz had no components.

Everything would be inside one folder.

```
Java Files

XML Files

Entities

Services

Widgets

Data

Images

JavaScript
```

Imagine thousands of files inside one folder.

It would become impossible to maintain.

So OFBiz divides the project into Components.

Each Component is responsible for one business domain.

---

# Example

Order Component

Responsible for

- Orders
- Order Items
- Payments
- Shipments

Product Component

Responsible for

- Products
- Categories
- Inventory
- Product Features

Party Component

Responsible for

- Customers
- Organizations
- Employees
- Contact Mechanisms

Every Component has its own responsibility.

---

# Real Component Structure

A typical component looks like this:

```
plugins/example/

│

├── config/

├── data/

├── documents/

├── dtd/

├── entitydef/

├── lib/

├── script/

├── servicedef/

├── src/

├── testdef/

├── webapp/

├── widget/

├── build.xml

└── ofbiz-component.xml
```

Every folder has a different purpose.

Let's understand each one.

---

# 1. config/

Purpose:

Stores configuration files.

Examples:

```
ExampleUiLabels.xml

example.properties
```

Usually contains

- UI Labels
- Properties Files
- Configuration

---

## ExampleUiLabels.xml

Purpose:

Supports

- Internationalization (i18n)
- Localization (l10n)

Example

```
First Name

↓

English

First Name

↓

French

Prénom

↓

German

Vorname
```

Instead of hardcoding text,

we store labels here.

---

## .properties Files

Stores key-value pairs.

Example

```
application.name=Example

company.name=HotWax
```

These values can be reused throughout the application.

---

# 2. data/

Purpose:

Contains XML data that will be loaded into the database.

Examples

- Seed Data
- Seed Initial Data
- Demo Data
- Ext Data

---

Example

```
ExampleTypeData.xml

ExampleDemoData.xml
```

Whenever we run

```
./gradlew load
```

the files inside this folder are loaded according to the configured data readers.

---

# 3. entitydef/

Purpose:

Contains Entity Definitions.

Example

```xml
<entity entity-name="Example">

    <field name="exampleId"
           type="id"
           is-pk="true"/>

    <field name="exampleName"
           type="name"/>

</entity>
```

This folder defines

- Tables
- Fields
- Primary Keys
- Relationships

---

# 4. servicedef/

Purpose:

Defines Services.

Example

```xml
<service

name="createExample"

engine="java">
```

Here we define

- IN Parameters
- OUT Parameters
- Authentication
- Transactions
- Engine

This folder only defines services.

Actual logic is written elsewhere.

---

# 5. script/

Purpose:

Contains Groovy scripts.

Examples

```
CreateOrder.groovy

CreateCustomer.groovy

Inventory.groovy
```

Whenever business logic is written in Groovy,

it usually lives here.

---

# 6. src/

Purpose:

Contains Java source code.

Example

```
events/

services/

helper/

util/
```

Usually contains

- Java Services
- Events
- Helper Classes
- Utility Classes

---

# 7. widget/

Purpose:

Contains UI XML.

Examples

```
ExampleScreens.xml

ExampleForms.xml

ExampleMenus.xml

ExampleTrees.xml
```

Responsible for rendering UI.

---

# 8. webapp/

Purpose:

Contains everything related to the Web Application.

Example

```
webapp/

│

├── WEB-INF/

│

├── images/

├── css/

├── js/

├── templates/
```

---

Inside WEB-INF

```
controller.xml

web.xml
```

These files control request routing.

We'll study them in detail later.

---

# 9. documents/

Purpose:

Stores documentation related to the component.

Usually

- PDFs
- Notes
- Specifications

Not every component contains this folder.

---

# 10. dtd/

Purpose:

Stores DTD files.

DTD means

```
Document Type Definition
```

Earlier XML validation was done using DTD.

Nowadays,

XSD is mostly used.

---

# 11. lib/

Purpose:

Stores component-specific libraries.

Example

```
my-library.jar
```

If only one component needs a library,

we place it here.

---

# 12. testdef/

Purpose:

Contains test definitions.

Used for automated testing.

Not commonly modified during day-to-day development.

---

# 13. build.xml

Purpose:

Component-specific build instructions.

Sometimes used for

- Custom Gradle Tasks
- Build Automation

Not every component requires customization here.

---

# 14. ofbiz-component.xml

This is the MOST IMPORTANT file inside every component.

Think of it as the **entry point of the component**.

Whenever OFBiz loads a component,

it first reads:

```
ofbiz-component.xml
```

---

# Why is ofbiz-component.xml important?

Because it tells OFBiz

"What resources belong to this component?"

Examples

- Entity Definitions
- Service Definitions
- Data Files
- Web Applications
- Widgets
- Classpath Resources

Without this file,

OFBiz doesn't know what to load.

---

```

↓

↓

```

Examples

# Component Relationship

# Chapter 4 - OFBiz Request Lifecycle (End-to-End Flow)

---

# Why do we need to understand Request Lifecycle?

Whenever a user clicks a button,

or opens any page,

or submits any form,

a request travels through many components before the response is returned.

If you understand this flow,

then debugging becomes very easy.

Whenever you encounter any issue,

you'll immediately know where to look.

---

# Complete Request Lifecycle

```
Browser

↓

HTTP Request

↓

Tomcat

↓

web.xml

↓

ControlServlet

↓

RequestHandler

↓

controller.xml

↓

request-map

↓

Event

↓

Service

↓

Entity Engine

↓

SQL

↓

Database

↓

Result

↓

View Map

↓

Screen Widget

↓

FTL

↓

HTML Response

↓

Browser
```

This is the complete lifecycle of one request.

Every page inside OFBiz follows this flow.

---

# Step 1 — Browser

Everything starts from the Browser.

Example

```
http://localhost:8443/webtools/control/main
```

or

```
http://localhost:8443/order/control/main
```

The browser creates an HTTP Request.

Types of requests:

- GET
- POST
- PUT
- DELETE

Normally,

opening a page uses GET.

Submitting a form usually uses POST.

---

# Step 2 — HTTP Request

The browser sends an HTTP Request.

The request contains:

- URL
- Headers
- Parameters
- Cookies
- Session ID
- Request Body

Example

```
GET

/order/control/main
```

---

# Step 3 — Tomcat

The request reaches Tomcat.

Tomcat is the Web Server (Servlet Container).

Tomcat does NOT understand OFBiz.

Tomcat only knows:

"I have received an HTTP Request."

Now it needs to decide:

Which application should handle this request?

---

# Step 4 — web.xml

Tomcat reads

```
WEB-INF/web.xml
```

This file is called the **Deployment Descriptor**.

Its responsibility is:

- Register Servlets
- Register Filters
- Register Listeners
- Define Context Parameters

Most importantly,

it tells Tomcat

which Servlet should receive incoming requests.

---

# Step 5 — ControlServlet

After reading web.xml,

Tomcat forwards the request to

```
ControlServlet
```

Think of ControlServlet as the **main entry point of OFBiz**.

Every request first enters here.

ControlServlet itself doesn't contain business logic.

Its job is simply:

Receive Request

↓

Pass it to RequestHandler

---

# Step 6 — RequestHandler

RequestHandler is responsible for understanding

"What does this URL mean?"

Example

```
/order/control/findOrders
```

RequestHandler now needs to find

```
findOrders
```

inside

```
controller.xml
```

---

# Step 7 — controller.xml

This file contains routing information.

Example

```xml
<request-map uri="findOrders">

    ...

</request-map>
```

Think of controller.xml like Google Maps.

It decides

"Where should this request go?"

It contains:

- request-map
- view-map
- event
- security
- response

---

# Step 8 — request-map

Suppose the request is

```
findOrders
```

controller.xml searches for

```xml
<request-map uri="findOrders">
```

Inside request-map,

OFBiz decides

Should it call:

- Java Event
- Service
- Screen

This decision is made here.

---

# Step 9 — Event

An Event receives

```
HttpServletRequest

HttpServletResponse
```

Its responsibilities:

- Read Parameters
- Validate Input
- Perform Simple Logic
- Call Services

Event should NOT contain heavy business logic.

Its main purpose is

Request Handling.

---

# Event Signature

Example

```java
public static String createOrder(

HttpServletRequest request,

HttpServletResponse response)
```

Event always returns

```
String
```

Examples

```
success

error

login
```

---

# Step 10 — Service

The Event usually calls a Service.

Service is responsible for

Business Logic.

Example

```
Approve Order

Reserve Inventory

Create Shipment

Capture Payment
```

Unlike Event,

Service does NOT know anything about

HttpServletRequest.

Service only works with

```
Map<String,Object>
```

Input

↓

Business Logic

↓

Map<String,Object>

Output

---

# Step 11 — Entity Engine

Whenever the Service needs data,

it calls the Entity Engine.

Example

```
Find Product

Find Customer

Update Order

Create Shipment
```

Entity Engine converts

Entity Query

↓

SQL

---

# Step 12 — SQL Generation

Entity Engine prepares SQL.

Example

```
EntityQuery

↓

SELECT

↓

FROM

↓

WHERE

↓

ORDER BY

↓

LIMIT
```

SQL is prepared automatically.

Developer usually doesn't write raw SQL.

---

# Step 13 — Database

SQL reaches the Database.

Example

```
MySQL

PostgreSQL

Derby
```

Database executes SQL

and returns ResultSet.

---

# Step 14 — Entity Engine receives Result

Entity Engine converts

```
ResultSet

↓

GenericValue

↓

EntityList
```

Now Java objects are returned

instead of database rows.

---

# Step 15 — Service returns Result

Service prepares

```
Map<String,Object>
```

Example

```
order

customer

totalAmount

status
```

Everything is stored inside one Map.

---

# Step 16 — Event receives Result

Event receives the Map.

Now Event returns

```
success
```

or

```
error
```

---

# Step 17 — view-map

controller.xml checks

```xml
<response>

name="success"

type="view"

value="OrderList"
```

Now OFBiz knows

which Screen should be rendered.

---

# Step 18 — Screen Widget

Example

```
OrderScreens.xml
```

Screen prepares UI.

Inside Screen

```
Actions

↓

Widgets

↓

Forms

↓

Menus
```

---

# Step 19 — FTL

Screen Widget renders

```
FreeMarker Template
```

FTL converts data into HTML.

Example

```
Customer Name

Order Number

Products

Tables
```

---

# Step 20 — HTML Response

HTML is returned to the Browser.

The Browser displays the page.

The request lifecycle is now complete.

---

# Complete Sequence Diagram


# Typical Event Structure

```java
public static String createOrder(

HttpServletRequest request,

HttpServletResponse response)

{

    // Read Request

    String productId

    = request.getParameter("productId");



    // Validation

    if(UtilValidate.isEmpty(productId))

    {

        return "error";

    }



    // Prepare Context

    Map<String,Object> ctx

    = UtilMisc.toMap(

        "productId",

        productId);



    // Call Service

    dispatcher.runSync(

    "createOrder",

    ctx);



    return "success";

}
```

Notice

Event never writes SQL.

# Chapter 12 - Entity Engine Internals (How OFBiz Talks to the Database)

---

# Goal of this Chapter

> **How does the Entity Engine itself work internally?**

We'll follow a query from the moment a developer writes Java code until the database returns the result.

This chapter focuses on the **internal architecture** of the Entity Engine.

---

# What is the Entity Engine?

The Entity Engine is one of the core frameworks of OFBiz.

Its job is to hide database-specific implementation from developers.

Instead of writing

```sql
SELECT *
FROM PRODUCT;
```

Developers write

```java
EntityQuery.use(delegator)
    .from("Product")
    .queryList();
```

The Entity Engine takes care of everything else.

---

# Responsibilities of Entity Engine

The Entity Engine is responsible for

- Reading Entity XML
- Creating entity metadata
- Building SQL
- Managing database connections
- Executing SQL
- Converting ResultSet into Java objects
- Managing entity cache
- Supporting multiple databases

Think of it as the **ORM layer** of OFBiz.

# Chapter 14 - Delegator (The Gateway to the Entity Engine)

---

# Goal of this Chapter

In previous chapters we learned

- Entity Engine
- EntityQuery
- EntityCondition
- GenericValue

But there is one object that connects all of them.

That object is

```
Delegator
```

Almost every Service, Event, and OFBiz component uses a Delegator.

If GenericValue represents **one database record**,

then Delegator represents **the gateway to the entire database**.

---

# What is a Delegator?

A Delegator is the primary interface used to interact with the Entity Engine.

It is responsible for

- Creating records
- Reading records
- Updating records
- Deleting records
- Looking up entity metadata
- Managing entity cache
- Delegating database work to the Entity Engine

Think of it as a **Database Manager**.

---

# Why is it called "Delegator"?

The name comes from the word

```
Delegate
```

Meaning

> "Pass the work to someone else."

Delegator itself does **not** execute SQL.

Instead it delegates the work to

```
Entity Engine

↓

SQL Builder

↓

JDBC

↓

Database
```

---

# High-Level Flow

```
Developer

↓

Delegator

↓

Entity Engine

↓

Database
```

Notice

Developer never talks directly to the Entity Engine.

Everything goes through the Delegator.

---

# Where does Delegator come from?

Inside a Service

```java
public static Map<String, Object> createProduct(
        DispatchContext dctx,
        Map<String, Object> context)
{
    Delegator delegator = dctx.getDelegator();

    ...
}
```

Inside an Event

```java
Delegator delegator =
request.getAttribute("delegator");
```

So every business logic already has access to a Delegator.

---

# Delegator vs GenericDelegator

Very important interview question.

```

↓

Interface
```

```
GenericDelegator

↓

Implementation
```

Think of it like

```

↓

ArrayList
```

or

```
Map

↓

HashMap
```

Similarly

```
Delegator

↓

GenericDelegator
```

Developers code against the interface.

The implementation performs the actual work.

---

# Why use an Interface?

Suppose tomorrow OFBiz introduces

```
AdvancedDelegator
```

Your code

```java

```

will continue to work.

This is one of the benefits of abstraction.

```

Transaction Coordination
```

Notice

Delegator is **not responsible for SQL generation**.

That work belongs to the Entity Engine.

# Chapter 15 - Transactions in OFBiz (Commit, Rollback & Data Integrity)

---

# Goal of this Chapter

Till now we have learned

- Entity Engine
- EntityQuery
- GenericValue
- Delegator

Now let's answer an important question.

> What happens if multiple database operations are running and one of them fails?

Example

```
Create Order

↓

Reserve Inventory

↓

Create Payment

↓

Create Shipment
```

Suppose Payment fails.

Should the Order still remain in the database?

**No.**

This is where **Transactions** come into the picture.

---

# What is a Transaction?

A Transaction is a **group of database operations that behave as one single unit**.

Meaning

```
Either

Everything succeeds

OR

Everything fails
```

There is no middle state.

---

# Example Without Transaction

Suppose we execute

```
Insert Order

✔ Success

↓

Reserve Inventory

✔ Success

↓

Create Payment

❌ Failed

↓

Create Shipment

Not Executed
```

Database becomes

```
Order Exists

Inventory Reserved

Payment Missing
```

This is an inconsistent database.

---

# Example With Transaction

Same flow

```
Insert Order

↓

Reserve Inventory

↓

Create Payment

↓

Shipment
```

Suppose Payment fails.

Result

```
Rollback

↓

Order Removed

↓

Inventory Released

↓

Database restored
```

Now database remains consistent.

---

# Why Transactions?

Transactions ensure

- Data Consistency
- Data Integrity
- No Partial Updates
- Safe Recovery

---

# Transaction Lifecycle

```
Begin Transaction

↓

Execute SQL

↓

Execute SQL

↓

Execute SQL

↓

Success?

↓

Yes

↓

Commit

OR

↓

Rollback
```

---

# Commit

Commit means

> Save all database changes permanently.

Flow

```
Transaction Started

↓

Insert Product

↓

Update Inventory

↓

Create Order

↓

Commit

↓

Saved Forever
```

---

# Rollback

Rollback means

> Undo every change made in the current transaction.

Flow

```
Transaction Started

↓

Insert Product

↓

Update Inventory

↓

Payment Failed

↓

Rollback

↓

Nothing Saved
```

---

# Visual Example

```
Transaction

{

Create Order

Reserve Inventory

Payment

Shipment

}
```

If Shipment fails

↓

Entire transaction rolls back.

---

# Real Example

Suppose customer places an order.

Business flow

```
Create Order Header

↓

Create Order Items

↓

Reserve Inventory

↓

Authorize Payment

↓

Generate Shipment
```

Suppose

```
Authorize Payment

↓

Fails
```

Result

```
Rollback Entire Transaction
```

Database becomes exactly as it was before.

---

# ACID Properties

Every transaction follows

```
ACID
```

---

# A - Atomicity

Meaning

```
All

OR

Nothing
```

Example

```
Bank Transfer

Withdraw ₹1000

Deposit ₹1000
```

Cannot happen partially.

---

# C - Consistency

Database should always remain valid.

Wrong

```
Order Exists

Payment Missing
```

Correct

```
Order

Payment

Shipment

Everything Consistent
```

---

# I - Isolation

Multiple users can perform transactions simultaneously.

Example

```
User A

Creating Order

||

User B

Updating Product
```

They should not interfere with each other.

---

# D - Durability

Once committed

data survives

```
Server Restart

Power Failure

Crash
```

because it is stored in the database.

---

# What is a View Entity?

A View Entity is a **virtual entity** that combines data from multiple entities.

Think of it as

```
Multiple Entities

↓

One Virtual Entity

↓

One Query
```

Unlike normal entities,

a View Entity does **not** create a new database table.

It only defines

how multiple tables should be joined.

---

### View Entity

Represents

```
Many Tables

↓

One Virtual View
```

Example

```

+

OrderItem

+

Product

=

OrderProductView
```

# Chapter 21 - services.xml (Where Every OFBiz Service is Defined)

---

# Goal of this Chapter

In the previous chapter, we learned

```
ModelService
```

Now let's answer another important question.

> Where does ModelService come from?

The answer is

```
services.xml
```

Every service that exists in OFBiz is first declared inside

```
services.xml
```

During server startup,

OFBiz reads this file,

creates ModelService objects,

and stores them in memory.

---

# High-Level Flow

```
Developer writes

↓

services.xml

↓

Service Reader

↓

ModelService

↓

Dispatcher

↓

Service Execution
```

Notice

```
services.xml

does NOT execute anything.
```

It only defines

how a service should behave.

---

# What is services.xml?

It is an XML configuration file

that describes

- Service Name
- Engine
- Java Class
- Method
- Parameters
- Authentication
- Transactions
- Validation

Think of it as

```
Blueprint

for every service.
```

---

# Location of services.xml

Usually found inside

```
component/

↓

servicedef/

↓

services.xml
```

Example

```
applications/

↓

order/

↓

servicedef/

↓

services.xml
```

Every component can have

its own

```
services.xml
```

---

# Basic Structure

```xml
<services>

    <service ...>

    </service>

</services>
```

The root tag

is

```
<services>
```

Inside it,

multiple

```
<service>
```

tags exist.

---

# Example

```xml
<service

name="createOrder"

engine="java"

location="org.apache.ofbiz.order.OrderServices"

invoke="createOrder"

auth="true"/>
```

This does NOT execute

```
createOrder()
```

It only tells OFBiz

where it exists.

---

# Service Lifecycle

```
Developer

↓

↓

Server Starts

↓

Read XML

↓

Create ModelService

↓

Dispatcher Uses It

↓

Java Executes
```

---

# The `<service>` Tag

Every service begins with

```xml
<service ...>
```

Example

```xml
<service

name="approveOrder"

engine="java"

location="..."

invoke="approveOrder"/>
```

---

# name

Example

```xml
name="createShipment"
```

This is the

unique identifier.

Dispatcher later calls

```java
dispatcher.runSync(
        "createShipment",
        context);
```

Notice

Dispatcher searches

using

```
name
```

---

# engine

Example

```xml
engine="java"
```

Possible values

```
java

groovy

simple

entity-auto
```

Engine tells OFBiz

how the service

should be executed.

---

# location

Example

```xml
location="org.apache.ofbiz.order.OrderServices"
```

Meaning

Java implementation

is located here.

---

# invoke

Example

```xml
invoke="createOrder"
```

Dispatcher finally calls

```java
OrderServices.createOrder()
```

---

# auth

Example

```xml
auth="true"
```

Meaning

User must be logged in

before execution.

If

```
auth="false"
```

Anonymous users

may invoke the service

(subject to the application's exposure and security configuration).

---

# use-transaction

Example

```xml
use-transaction="true"
```

Meaning

Dispatcher automatically

starts a transaction.

Flow

```
Begin Transaction

↓

Execute Service

↓

Commit

or

Rollback
```

---

# export

Example

```xml
export="true"
```

Meaning

Service is available

to remote clients

through configured service interfaces.

---

# validate

Some services define

```xml
validate="true"
```

Meaning

Dispatcher validates

parameters

before execution.

---

# debug

Sometimes you'll see

```xml
debug="true"
```

Used mainly

during development

to assist debugging.

---

# mode

Possible values


---

## IN

Caller provides value.

Example

```xml

```

---

## OUT

Service returns value.

Example

```xml

```

---

## INOUT

# optional

Example

```xml

```

Meaning

Dispatcher checks

before execution.

# Default Values

# Example Service

```xml
<service

name="createProduct"

engine="java"

location="org.apache.ofbiz.product.ProductServices"

invoke="createProduct"

auth="true"

use-transaction="true">

    <attribute

        name="productId"

        type="String"

        mode="IN"/>

    <attribute

        name="internalName"

        type="String"

        mode="IN"/>

    <attribute

        name="product"

        type="GenericValue"

        mode="OUT"/>

</service>
```

This definition tells OFBiz

everything required

to execute

```
createProduct()
```

# Common Mistakes

## Mistake 1

Thinking

services.xml

contains business logic.

Wrong.

Business logic exists

inside

Java,

Groovy,

or Simple services.

---

## Mistake 2

Changing

Java method name

but forgetting

to update

```
invoke
```

Dispatcher

won't find

the method.

---

## Mistake 3

Defining incorrect

parameter types.

Validation

will fail

before execution.

---

## Mistake 4

Forgetting to define

mandatory input parameters

using `<attribute>`.

The service may receive incomplete data or fail validation.

---

# Best Practices

✔ Keep one logical group of services per `services.xml`.

✔ Give services clear, action-oriented names (`createOrder`, `cancelShipment`, `approveInvoice`).

✔ Define all input and output parameters explicitly.

✔ Use `use-transaction="true"` for operations that modify multiple entities.

✔ Keep XML metadata synchronized with the actual implementation.
